# 性能优化记录 (mikuro_mod_packer)

这是一份"逐项较真"的优化流水账:每改一处,都把 before/after 基准和**字节一致校验**记下来。机器是 16 核 (Win11)。
方法很笨但可靠——每项都用 `git stash` 拿 old/new 两份代码跑同一负载对比,再逐字节比对输出,确认提速没改结果。

> **「字节一致」列怎么读**: `N/N 一致` = N 个样本全部逐字节相同(0 处不同)= 完美。
> 列里的数字是匹配数,不是差异数;✅ = 输出跟优化前一模一样,纯提速、不动结果。这是我给每项优化定的硬底线:**字节一致才算数,否则只是碰运气**。

> 这份记录开始之前已经做掉的两项: MPP numpy 向量化 `build_grid_np` (9f71e0b, 单 tile ~1.8x, 144/144 字节一致);
> 并行化离线 MPP 生成 (45a88a3, 48-tile 批 3.1x)。下面从 #1 接着记。

---

## #1 mesh 几何跨 tile 缓存

**问题**: `native.gather` 的 mesh 缓存是每 tile 新建 (`cache={}`), 同一 `.mesh` 被每个用到它的 tile 反复 `load_mesh_file` 解析一遍。`Context.MeshBoundsCache` 只缓存 bounds, 不缓存几何。

**做法**: 把几何缓存挂到 `Context.mesh_geom_cache` (并行 worker 之间复用), `gather` 改用它。

| 负载 | old | new | 加速 | 字节一致 |
|---|---|---|---|---|
| 40 大 tile / 单 ctx | 82.31s | 82.32s | **1.00x** | ✅ 40/40 一致 (0 处不同) |

**结论(说实话)**: 现在就是空转——mesh 解析(`gather`≈0.006s/tile)跟 classify(~2s/大 tile)比可以忽略不计,缓存那点收益直接被淹了。**但我故意把它留着给 #4 铺路**: 等 numba 把 classify 压下去 50-100x 之后, mesh 解析就会浮上来成主成本, 那时候这个跨 tile 缓存才真正派上用场。改动小、字节一致、是个前瞻投资,留。

---

## #2 pack_mod 并行压缩

**问题**: `pack_mod` 的 `zlib.compress`/`crc32` 是串行逐 block 一个个啃。

**做法**: 抓住一个关键事实——`zlib.compress` 是**释放 GIL** 的 → 直接上 `ThreadPoolExecutor`(零 pickling, 线程之间共享 `overrides`/content)。三趟走: Pass A 按序读 content, Pass B 并行压, Pass C 严格按 manifest 序组装(offset 依赖累积位置, 这趟不能乱)。

| 负载 | old | new | 加速 | 字节一致 |
|---|---|---|---|---|
| 挑战者大陆--POE (18.5MB) | 2.32s | 1.54s | **1.51x** | ✅ sha 一致 |
| MIKURO_CLASS_QLJX (1.2MB) | 0.54s | 0.47s | 1.15x | ✅ |
| MIKURO_VANILLA (2.2MB) | 0.99s | 0.95s | 1.04x | ✅ |
| 小 mod (<64 block 走串行) | — | — | ~1.0x(噪声) | ✅ |

**结论**: 大 mod 上 1.51x(5 个 mod 的 sha256 全逐字节一致)。卡在 Amdahl 上——压缩只占 pack 一半, 另一半是文件扫描/读盘/写出(I/O 难并行)。mod 越大、压缩占比越高 → 收益越大。线程方案零 pickling、风险低, 这种"白捡的"先拿。

## #3 rg_hash 缓存

**问题**: `rg_hash` 没 memo, 同一个 key 在序列化里被算成千上万遍。

**做法**: 套一层 `@functools.lru_cache(maxsize=None)`(只有 bindat 用 rg_hash; key/section 名重复率接近 1、且有界, 不会无限涨)。

| 负载 | old | new | 加速 | 字节一致 |
|---|---|---|---|---|
| 400 DAT compile | 0.170s | 0.159s | **1.07x** | ✅ 399/399 一致 (0 处不同) |

**结论**: 小赚一点。rg_hash 本来就只占 compile_dat 一小块(~0.5ms/file), 而且 key 短, lru_cache 的查表开销几乎把哈希省下的又吃回去。但它字节一致、就一行、无副作用(输入有界不增长), 没理由不留。

---

## #4 MPP numba @njit

**问题**: classify 是一堆标量 f64 紧循环, Python 也好 numpy 也好都到顶了。

**做法**: `mpp/native_nb.py` 把那些标量核(point_in_tri/closest_on_tri/swept_tri + 下射线/clearance/enclosure)一条运算一条运算地镜像进 `@njit(fastmath=False)`(显式 mul、保持同序、禁掉 FMA/重排 → IEEE754 逐位一致)。`gather`(mesh 解析+变换)还是纯 Python。`build_grid_nb` 把数据转成 numpy 数组后再喂核。numba 装不上时 pipeline 自动回退 `build_grid_np`。

动手前先做了道保险: `@njit(fastmath=False)` vs Python 浮点 **0/300000 逐位一致** —— 确认 numba 不会悄悄改变浮点结果, 才敢往上压。

| 负载 | old(np) | new(nb) | 加速 | 字节一致 |
|---|---|---|---|---|
| classify only (8 大 tile) | 9.29s | 0.254s | **36.5x** | ✅ 134/134 一致 (nb vs np/build_grid) |
| compile_mpp 端到端 (60 大 tile) | 91.70s | 6.81s | **13.5x** | ✅ 57/57 一致 (0 处不同) |

**结论**: numba @njit(fastmath=False) 字节一致, classify 干到 **36.5x**(单 tile 22-42x)。端到端 13.5x——又是 Amdahl: classify 不再是瓶颈了, 现在轮到 **header 解析 + gather** 当主成本(这下 #1 的 mesh 缓存终于开始生效, 前瞻没白留)。再叠上原有的并行 MPP 生成。numba 0.65.1 跟 numpy 2.4.6 能和平共处、不用降级; 首次 JIT 编译 ~1-2s, 之后 `cache=True` 从盘加载。

## #5 layout 解析 (numba 之后的端到端瓶颈调查)

numba 上了之后再 profile compile_mpp (30 tile/5.17s): **parse_layout 2.60s(占 50%)** / build_grid_nb 含 gather 1.52s / compute_region 1.02s。拆开 parse_layout 内部 = 文本 tokenize(正则/strip/18×`_fget`)+ **transform 数学**(`world`+`local_matrix` ~1.3s,90% 的对象都得算)+ `_transform_bounds_f32` 0.42s。

**做了的(都是字节一致的清理, 不当性能主张)**:
- W1: 删掉 `_parse_baseobject` 里那个死字典 `multi`(逐 prop setdefault/append 一通, 然后从来没人用)。
- W2: `build_grid_nb` 里 tris→numpy 从 9·n 逐元素赋值改成一次 `np.array([t[:3] for t])`。

| 负载 | old | new | 加速 | 字节一致 |
|---|---|---|---|---|
| compile_mpp 端到端 (60 大 tile) | 7.03s | 6.91s | **1.02x** | ✅ 57/57 一致 |

**结论(说实话)**: 就 1.02x——layout 解析是均摊在 CPU 上的杂活(正则/strip/_fget 文本 + 矩阵变换数学), 微优化全被 wash 掉(跟 BINLAYOUT 一个德行)。W1 是删死代码(本来就该删)、W2 更干净, 字节一致就留着, 但**不拿它当性能成绩**。真要撬动, 得把 transform 数学(world/local_matrix/_transform_bounds_f32 ~1.7s)整个搬进 numba——可那是在 Python 对象树上动大手术、有字节风险、估计端到端也就 ~1.25x, **ROI 太低**。#4 已经把 MPP 的大头(13.5x)吃掉了, 这儿已经贴近纯 Python 的实际下限, 不值得再深挖。

## #6 全量打包流水线 (2026-06-20: 正确性修复 + BINDAT/RAW/Pack 并行 + 算法修复)

> 全量基准 `tools/bench_all_mods.py`: 把全部 25 个安装 mod **从零打包**, 逐阶段计时 (Compile/RAW/MPP/Pack)。16 核, 纯内存、非破坏。全部**逐字节 / 语义一致**, 90 测试全过, 全库 0 错误。

这一轮是从"能跑"迈到"跑得快"的关键一跳, 也是顺手把两个进游戏的正确性 bug 一起修掉的一轮。**全量总计的逐步下降:**

| 节点 | 总计 | RAW | Pack | 关键改动 |
|---|--:|--:|--:|---|
| 起点(正确性修完后) | 185.5s | 35.2 | 93.5 | hash BINDAT + RAW 并行 + rollingHash + 大小写修复 |
| + Pack 第一轮 | 158.5s | 40.2 | 54.0 | isal + scandir + 压缩线程 8→16 |
| + O(n²) manifest | 107.0s | 34.5 | 15.8 | `bytes +=` → list+`b''.join` |
| **+ RAW scandir+线程** | **83s** | **18.4** | 14.4 | 本轮 |

**逐项(每项都字节一致 / 语义一致)**:

- **BINDAT per-file hash 并行** (62f0a1c): 把全局 corpus 字典(`StringDict` 自增计数器)整个扔了, 换成 `HashStringDict`(`rg_hash`+文件内探测)。底气来自一个发现: 游戏是逐文件解析 id 的(已证: shipped base game 自己就带 565 处跨文件撞号还照常加载; 而且 hash 包进游戏后职业/技能/图标全对)→ 没有共享状态 → 单遍进程池就行。15k-DAT 的 mod **5.69→1.59s (3.59x)**, 而且 serial==parallel。
- **RAW scandir** (dcdeae9): `_scan`/`_scan_interleaved` 把每文件一次 `os.path.isdir` stat 换成 `os.scandir`(DirEntry 自带 is_dir, 省掉一次 stat)。AFFIXES scan **3.02→0.07s**, 这种几十倍的收益基本是白捡的。
- **RAW 线程池+共享缓存** (ac44ac4): 重 builder 是读盘密集型(几千个小 UTF-16 DAT), 进程池的 IPC 反而拖后腿 → 改 `ThreadPoolExecutor`(I/O 放 GIL、零 pickling、UNITDATA 继承链缓存还能跨线程共享)。职业技能 RAW **9.93→3.91s**。
- **Pack O(n²)→O(n) manifest** (b3601d2): `_w_manifest` 那句 `out += bytes` 是经典 O(n²) 拷贝, 44k 记录硬生生跑 16.5s → 改 list + `b''.join`(~0.1s)。地图拓展 pack **20.89→1.84s (11x)**。这个 bug 藏得最深, 收益也最大。
- **Pack scandir** (62f0a1c): manifest 扫描同样把 listdir+stat 换 scandir。68k-file 的 mod, manifest **14.79→0.93s (16x)**。
- **Pack isal+16 线程** (62f0a1c): `isal_zlib`(SIMD DEFLATE, 仍是 zlib 格式, 游戏照常解压; 装不上回退 zlib)+ 压缩线程 8→16。字节重的 mod(贴图多)压缩 **~4.9x**(代价是体积 +~5%)。

**顺带修掉的两个进游戏正确性 bug**: ① rollingHash 游戏是会校验的(stride RNG 拿数据长度当种子→确定性), 写 0 就静默 "Unable to load mod"; ② manifest 文件名 GUTS 要全大写, 我们存了磁盘上的小写 → `.dds` 查不到 → 职业头像显示错图。这两个的完整来龙去脉见 [`EditorGuts逆向-MOD打包与读取-完整记录.md`](EditorGuts逆向-MOD打包与读取-完整记录.md) 的 §1.5 / §1.6。

**结论**: 全量从 **185.5s 压到 ~82s(~2.2x)**(那次定格在 83s, 之后复跑在 81.6–84s 之间晃, 取 ~82s)。最大的隐藏收益其实不在压缩, 而在 Pack 的 O(n²) manifest 写入, 加上文件数大的 mod 的 scandir——都是"算法/系统调用"层面的, 不是算力层面的。现在 4 个阶段(Compile 24 / RAW 18 / MPP 26 / Pack 14)基本均衡、各自并行, 进入递减区了。还试过把 (Compile‖RAW‖MPP) 三段并发跑(`tools/perf_overlap.py`): 字节一致, 但被 CPU 饱和死死封在 ~1.28x, ROI 不够, 没实装。
