# 性能优化记录 (mikuro_mod_packer)

逐项性能优化的 before/after 基准 + 字节一致校验记录。机器: 16 核 (Win11)。
方法: 每项用 `git stash` 对比 old/new 代码同一负载, 并校验输出逐字节一致。

> **「字节一致」列读法**: `N/N 一致` = N 个样本全部逐字节相同(0 处不同)= 完美。
> 数字是匹配数, 不是差异数; ✅ = 输出与优化前完全相同, 优化纯属提速、不改结果。

> 已完成(本记录之前): MPP numpy 向量化 `build_grid_np` (9f71e0b, 单 tile ~1.8x, 144/144 字节一致);
> 并行化离线 MPP 生成 (45a88a3, 48-tile 批 3.1x)。

---

## #1 mesh 几何跨 tile 缓存

**问题**: `native.gather` 的 mesh 缓存是每 tile 新建 (`cache={}`), 同一 `.mesh` 被每个用到它的 tile 重复 `load_mesh_file` 解析。`Context.MeshBoundsCache` 只缓存 bounds, 不缓存几何。

**做法**: 几何缓存挂到 `Context.mesh_geom_cache` (并行 worker 复用), `gather` 改用它。

| 负载 | old | new | 加速 | 字节一致 |
|---|---|---|---|---|
| 40 大 tile / 单 ctx | 82.31s | 82.32s | **1.00x** | ✅ 40/40 一致 (0 处不同) |

**结论(坦诚)**: 现在是空转——mesh 解析(`gather`≈0.006s/tile)相对 classify(~2s/大 tile)可忽略,缓存收益被淹没。**但这是为 #4 铺路**: numba 把 classify 压 50-100x 后, mesh 解析会变成主成本, 届时这个跨 tile 缓存才生效。改动小、字节一致、前瞻性保留。

## #2 pack_mod 并行压缩

**问题**: `pack_mod` 的 `zlib.compress`/`crc32` 是串行逐 block 循环。

**做法**: `zlib.compress` **释放 GIL** → 用 `ThreadPoolExecutor`(零 pickling, 线程共享 `overrides`/content)。Pass A 按序读 content, Pass B 并行压缩, Pass C 严格按 manifest 序组装(offset 依赖累积位置)。

| 负载 | old | new | 加速 | 字节一致 |
|---|---|---|---|---|
| 挑战者大陆--POE (18.5MB) | 2.32s | 1.54s | **1.51x** | ✅ sha 一致 |
| MIKURO_CLASS_QLJX (1.2MB) | 0.54s | 0.47s | 1.15x | ✅ |
| MIKURO_VANILLA (2.2MB) | 0.99s | 0.95s | 1.04x | ✅ |
| 小 mod (<64 block 走串行) | — | — | ~1.0x(噪声) | ✅ |

**结论**: 大 mod 上 1.51x(全 5 mod sha256 逐字节一致)。受 Amdahl 限制——压缩约占 pack 一半, 另一半是文件扫描/读盘/写出(I/O 难并行)。mod 越大压缩占比越高 → 收益越大。线程方案零 pickling、低风险。

## #3 rg_hash 缓存

**问题**: `rg_hash` 无 memo, key 在序列化里重复成千上万次。

**做法**: `@functools.lru_cache(maxsize=None)`(仅 bindat 用 rg_hash; key/section 名重复率 ~1、有界)。

| 负载 | old | new | 加速 | 字节一致 |
|---|---|---|---|---|
| 400 DAT compile | 0.170s | 0.159s | **1.07x** | ✅ 399/399 一致 (0 处不同) |

**结论**: 小正收益。rg_hash 只占 compile_dat 一小部分(本就 ~0.5ms/file), 短 key 下 lru_cache 查找开销几乎抵消哈希节省。但字节一致、一行改动、无副作用(输入有界不增长), 保留。

## #4 MPP numba @njit

**问题**: classify 标量 f64 紧循环, Python/numpy 都有上限。

**做法**: `mpp/native_nb.py` 把标量核(point_in_tri/closest_on_tri/swept_tri + 下射线/clearance/enclosure)逐运算镜像进 `@njit(fastmath=False)`(显式 mul、同序、无 FMA/重排 → IEEE754 逐位一致)。`gather`(mesh 解析+变换)仍纯 Python。`build_grid_nb` 转 numpy 数组后调核。numba 缺失时 pipeline 回退 `build_grid_np`。

前置验证: `@njit(fastmath=False)` vs Python 浮点 **0/300000 逐位一致**。

| 负载 | old(np) | new(nb) | 加速 | 字节一致 |
|---|---|---|---|---|
| classify only (8 大 tile) | 9.29s | 0.254s | **36.5x** | ✅ 134/134 一致 (nb vs np/build_grid) |
| compile_mpp 端到端 (60 大 tile) | 91.70s | 6.81s | **13.5x** | ✅ 57/57 一致 (0 处不同) |

**结论**: numba @njit(fastmath=False) 字节一致, classify **36.5x**(单 tile 22-42x)。端到端 13.5x——Amdahl: classify 不再是瓶颈, 现在 **header 解析 + gather** 成主成本(故 #1 mesh 缓存在此开始生效)。叠加已有并行 MPP 生成。numba 0.65.1 与 numpy 2.4.6 共存无需降级; 首次 JIT 编译 ~1-2s, `cache=True` 后续从盘加载。

## #5 layout 解析 (numba 后的端到端瓶颈调查)

numba 之后 compile_mpp profile (30 tile/5.17s): **parse_layout 2.60s(50%)** / build_grid_nb 含 gather 1.52s / compute_region 1.02s。parse_layout 内部 = 文本 tokenize(正则/strip/18×`_fget`)+ **transform 数学**(`world`+`local_matrix` ~1.3s,90% 对象都需要)+ `_transform_bounds_f32` 0.42s。

**做的(字节一致清理, 非性能主张)**:
- W1: 删 `_parse_baseobject` 里的死字典 `multi`(逐 prop setdefault/append 但从不使用)。
- W2: `build_grid_nb` 的 tris→numpy 从 9·n 逐元素赋值改为一次 `np.array([t[:3] for t])`。

| 负载 | old | new | 加速 | 字节一致 |
|---|---|---|---|---|
| compile_mpp 端到端 (60 大 tile) | 7.03s | 6.91s | **1.02x** | ✅ 57/57 一致 |

**结论(坦诚)**: 仅 1.02x——layout 解析是 CPU 均摊工作(正则/strip/_fget 文本 + 矩阵变换数学),微优化washout(同 BINLAYOUT)。W1 是删死代码(本就该删)、W2 更简洁,字节一致保留,但**不作性能主张**。真正的杠杆是把 transform 数学(world/local_matrix/_transform_bounds_f32 ~1.7s)搬进 numba——但那是在 Python 对象树上的大重构、字节风险、估计端到端仅 ~1.25x,**低 ROI**。#4 已经把 MPP 的大头(13.5x)拿下, 此处接近纯 Python 实际下限, 不建议深挖。

## #6 全量打包流水线 (2026-06-20: 正确性修复 + BINDAT/RAW/Pack 并行 + 算法修复)

> 全量基准 `tools/bench_all_mods.py`: 全部 25 个安装 mod **从零打包**, 逐阶段计时 (Compile/RAW/MPP/Pack)。16 核, 纯内存非破坏。全部**逐字节 / 语义一致**, 90 测试全过, 全库 0 错误。

**全量总计的逐步下降:**

| 节点 | 总计 | RAW | Pack | 关键改动 |
|---|--:|--:|--:|---|
| 起点(正确性修完后) | 185.5s | 35.2 | 93.5 | hash BINDAT + RAW 并行 + rollingHash + 大小写修复 |
| + Pack 第一轮 | 158.5s | 40.2 | 54.0 | isal + scandir + 压缩线程 8→16 |
| + O(n²) manifest | 107.0s | 34.5 | 15.8 | `bytes +=` → list+`b''.join` |
| **+ RAW scandir+线程** | **83s** | **18.4** | 14.4 | 本轮 |

**逐项(均字节一致 / 语义一致):**

- **BINDAT per-file hash 并行** (62f0a1c): 弃全局 corpus 字典(`StringDict` 自增计数器)改 `HashStringDict`(`rg_hash`+文件内探测)。游戏逐文件解析 id(已证: shipped base game 自带 565 处跨文件撞号仍正常加载; hash 包进游戏职业/技能/图标全对)→ 无共享状态 → 单遍进程池。15k-DAT mod **5.69→1.59s (3.59x)**, serial==parallel。
- **RAW scandir** (dcdeae9): `_scan`/`_scan_interleaved` 弃每文件 `os.path.isdir` stat 改 `os.scandir`。AFFIXES scan **3.02→0.07s**。
- **RAW 线程池+共享缓存** (ac44ac4): 重 builder 读盘密集(几千个小 UTF-16 DAT), 进程池 IPC 反拖累 → `ThreadPoolExecutor`(I/O 放 GIL、零 pickling、UNITDATA 链缓存跨线程共享)。职业技能 RAW **9.93→3.91s**。
- **Pack O(n²)→O(n) manifest** (b3601d2): `_w_manifest` 的 `out += bytes` 是 O(n²) 拷贝, 44k 记录=16.5s → list+`b''.join`(~0.1s)。地图拓展 pack **20.89→1.84s (11x)**。
- **Pack scandir** (62f0a1c): manifest 扫描同弃 listdir+stat。68k-file mod manifest **14.79→0.93s (16x)**。
- **Pack isal+16 线程** (62f0a1c): `isal_zlib`(SIMD DEFLATE, zlib 格式, 游戏照常解压; 缺失回退 zlib)+ 压缩线程 8→16。字节重 mod(贴图)压缩 **~4.9x**(+~5% 体积)。

**两个 in-game 正确性 bug(本轮同修)**: ① rollingHash 游戏会校验(stride RNG 用数据长度作种子→确定性), 写 0 → 静默 "Unable to load mod"; ② manifest 文件名 GUTS 全大写, 我们存了磁盘小写 → `.dds` 查不到 → 职业头像错图。详见 memory `mod-container-hash-count-fields-and-activation` / `mod-manifest-uppercase-filenames`。

**结论**: 全量 **185.5→83s (~2.2x)**。最大隐藏收益是 Pack 的 O(n²) manifest 写入 + 文件数重 mod 的 scandir。现在 4 阶段(Compile 24 / RAW 18 / MPP 26 / Pack 14)均衡且各自并行, 进入递减区。试验过 (Compile‖RAW‖MPP) 并发(`tools/perf_overlap.py`): 字节一致但被 CPU 饱和封在 ~1.28x, ROI 不足未实装。
